### Experiment: edge_pair_feat (content-based edge features)

**Hypothesis:** Adding content-based features (jaccard, containment, length ratio, position delta) to the edge pair interaction will break the position-based relation shortcut that causes long_decompose failures.

**Implementation:**
- Added `edge_pair_feat` field to `UnifiedBatch` dataclass (5 scalar features per (i,j) slot pair)
- Added `edge_pair_proj` two-layer MLP (5 → hidden_dim → hidden_dim) with zero-init on final layer so initial forward matches checkpoint baseline
- Edge-pair features are added elementwise to `left * right` interaction before concatenation into `pair_in`
- Features computed in `__getitem__`: jaccard similarity, containment_ij, containment_ji, char length ratio, normalized position delta

### Experiment: Curated Counterfactual Verifier (delta-preserving permutation)

**Hypothesis:** The previous `--counterfactual-rate` (random label permutation) failed because random permutation doesn't preserve position distribution — it creates unnatural relation→position mappings. Using **delta-preserving permutation** (only swap relations among edges with the same `j-i` delta) breaks the position shortcut while keeping position distribution identical.

**Implementation:**
- Modified `compute_loss` in `train_unified_v1.py`: existing `--counterfactual-rate` now uses delta-preserving permutation instead of random permutation
- For each counterfactual row, gold edges are grouped by delta `(j-i)`; within each delta group with ≥2 edges, relation labels are permuted
- This preserves: edge existence pattern, position-delta distribution, relation set per row
- This breaks: the `position(i,j) → relation` shortcut

**Motivating data:**
```text
Relation distribution at position (0,1): support=68.8%, part_of=22.2%, contradict=3.3%, ...
Relation distribution at position (1,2): part_of=44.6%, support=11.8%, contradict=5.1%, ...
The model learns "position (0,1) => support" as a shortcut (68.8% of training edges at (0,1) are "support").
```

**Command:**
```bash
python train_unified_v1.py \
  --train-jsonl artifacts/proposer_v1_20260512/proposer_train.jsonl \
  --val-jsonl   artifacts/proposer_v1_20260512/proposer_val.jsonl \
  --out-dir     out_unified_v1_verifier_curated_cf \
  --cand-emb-cache artifacts/spec_emb_cache/pred_v1_fix8_minilm_cand.npz \
  --mem-emb-cache  artifacts/spec_emb_cache/pred_v1_fix8_minilm_mem.npz \
  --resume-from out_unified_v1_warmstart_hardneg/best_unified_v1.pt \
  --epochs 4 --lr 2e-5 \
  --edge-exist-weight 1.2 --edge-rel-weight 0.5 \
  --verifier-weight 1.0 --freeze-backbone-for-edgepair \
  --counterfactual-rate 0.5 \
  --edge-rel-class-weight inverse_freq \
  --edge-rel-weight-min 0.5 --edge-rel-weight-max 3.0
```

**Success criteria:**
- long_decompose false_edge_rate improves from 0.833
- mixed_add_link preserved at 3/3
- multi_region_attach preserved at 2/2
- edge_precision improves or stays same

**Critical bug fix discovered during evaluation:**
The `out.get("verifier_logits", out["edge_rel_logits"])` pattern was a no-op because `verifier_logits` is ALWAYS in the output dict (model forward always computes it). All eval paths used `verifier_logits` even for baseline checkpoints where the verifier head was randomly initialized. Fixed in `eval_unified_roundtrip.py` and `train_unified_v1.py`:
- Changed to `out["verifier_logits"] if model.use_verifier else out["edge_rel_logits"]`
- Also fixed: edges with `rel == "none"` are now filtered from predicted_edges in metrics (was counting "none" as a valid edge relation, inflating false positives)
